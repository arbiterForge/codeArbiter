# Checkpoint — 2026-05-05 — Stage 1 (Prototype)

## Overview

This checkpoint covers a full triage of the FUSION Stage 1 scaffold as of 2026-05-05, spanning audit event correctness, trust-zone HTTP routing, test coverage compliance against CLAUDE.md §9 TDD Contract, CI/CD gate operability, scaffold completeness, and architectural decision health. The overall posture is **not promotion-ready**: 12 findings block Stage 2 entry, including two CRITICAL gaps in audit emission (no authn events emitted at all; emitted events missing required schema fields) and multiple HIGH findings covering bare-fetch trust-zone violations, absent Semgrep enforcement, inoperable local dev stack, and uncovered code paths that are direct TDD Contract violations. The Stage 1 scaffold is functionally complete for internal prototype use but carries meaningful compliance and safety debt that must be resolved before any non-team user or external environment is involved.

## Decision Challenges

### ADR-0001 — Adopt CLAUDE.md Contract

**Load-bearing assumption:** CLAUDE.md hard rules are enforced by live CI gates, not aspirational prose.

**Adversarial case:** `make up` is inoperable (F-010), `tools/agent_policy/` is absent (F-012), and verification signals (`make fips-check`, `make secrets-scan`, `make container-scan`) cannot execute end-to-end in CI. A solo maintainer under time pressure will treat failing gates as noise and begin treating CLAUDE.md §3 rules as advisory. Unresolved `[CONFIRM-NN]` items compound this by leaving policy gaps that agents must not fill by guessing.

**Confidence:** 2 / 5

**Hedge:** Produce a `make ci-dry-run` target that exits non-zero when required tooling (Semgrep rules, agent-policy linter, trivy, syft, cosign) cannot actually execute. Gate must be green before any Stage 2 promotion is considered.

---

### ADR-0002 — Adopt @xyflow/react@12

**Load-bearing assumption:** xyflow GmbH honors its MIT license pledge for the duration of FUSION's ATO timeline.

**Adversarial case:** xyflow GmbH is a small commercial company; "MIT forever" is a marketing statement, not a legal instrument. HashiCorp and Elasticsearch precedent demonstrates that commercial OSS maintainers relicense under business pressure. `@xyflow/system` has no independent CVE tracking surface, creating a hidden vulnerability surface.

**Confidence:** 3 / 5

**Hedge:** Document a drop-in alternative (e.g., `reactflow` fork, `elkjs`-based renderer) before graph UI features accumulate switching cost. Add a license-change tripwire to `make license-scan` that fails if `@xyflow/*` is no longer MIT.

---

### ADR-0003 — Adopt OCSF Audit Schema

**Load-bearing assumption:** The `audit.emit()` interface is transparently swappable across transports (HTTP → NATS → Kafka → WORM hash-chain) without changing calling code or delivery semantics.

**Adversarial case:** Delivery semantics differ materially: HTTP POST is synchronous and the caller knows immediately whether delivery succeeded; NATS is at-least-once async; Kafka adds consumer-group ordering and partition assignment. `[CONFIRM-05]` (S4 sink) is unresolved. Code written at S1 assuming synchronous emit will silently change correctness guarantees at S2 without any test catching the regression.

**Confidence:** 2 / 5

**Hedge:** Define the failure-semantics contract for `audit.emit()` explicitly as a test assertion now — not prose. The test must specify: what is the caller's observable behavior when the transport fails? This assertion becomes the invariant that transport swaps must preserve.

---

### ADR-0004 — Adopt Node.js/TypeScript Backend

**Load-bearing assumption:** Target classified/air-gapped environments have an npm mirror available, making "one registry" a simplification rather than a new single point of failure.

**Adversarial case:** Federal environments often have PyPI mirrors but not npm mirrors. The "one registry" argument inverts entirely if the target environment has no npm mirror. The ADR's secondary justification (AI-agent coherence) is a team property, not a deployed artifact property, and would not survive ATO review as a rationale standing alone.

**Confidence:** 3 / 5

**Hedge:** Validate the npm-mirror assumption against at least one named target customer environment before Stage 3 entry. Record the validation result (positive or negative) in `docs/decisions/0004` as an amendment.

## Risk Register Gaps

The following risks were identified during this review and are **not currently present in `docs/risks.md`**. New entries are warranted:

1. **xyflow license flip** — `@xyflow/react` relicenses away from MIT before FUSION reaches ATO. No tripwire in `make license-scan`. No documented drop-in alternative. Affects ADR-0002. (See F-042.)

2. **Audit transport semantic mismatch** — `audit.emit()` failure semantics are undefined as a machine-verifiable contract. Transport swap at S2+ (HTTP → NATS/Kafka) silently changes delivery guarantees without a failing test. Affects ADR-0003. [CONFIRM-05] unresolved. (See F-031.)

3. **npm mirror unavailability in classified environments** — Target air-gapped environments may have PyPI but not npm. The "one registry" simplification in ADR-0004 becomes a deployment blocker. No named target environment has been validated. Affects ADR-0004. (See F-032.)
